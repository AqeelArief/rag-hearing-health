# RAG Benchmarking for Hearing Healthcare Q&A

## Folder Structure

```
rag-hearing-health/
├── data/
│   ├── questions.csv          # your evaluation question set (you fill this in)
│   ├── corpus/                # raw source documents (PDFs/text you download)
│   └── corpus_metadata.csv    # metadata about each document
├── src/
│   ├── 01_build_corpus.py     # chunk documents, save chunk metadata
│   ├── 02_embed_and_index.py  # embed chunks, build vector stores per embedding model
│   ├── 03_run_experiment.py   # loop over all configs x questions, log results
│   ├── 04_score_answers.py    # LLM-as-judge scoring of correctness/hallucination
│   ├── 05_analyze_results.py  # statistical analysis (McNemar, logistic regression, bootstrap)
│   └── config.py              # central place to define your factorial grid
│   └── llm_client.py          # unified client for calling LLMs via Groq's API
├── results/
│   └── run_log.csv            # the big long-format results table (auto-generated)
├── figures/                   # output plots
├── notebooks/
│   └── eda_starter.py         # exploratory data analysis
└── requirements.txt
```

## Setup

```bash
# 1. Create and activate a virtual environment
python -m venv venv
source venv/bin/activate       # Windows: venv\Scripts\activate

# 2. Install dependencies
pip install -r requirements.txt

# 3. Get a free Groq API key and set it as an environment variable
# Sign up at https://console.groq.com/keys
export GROQ_API_KEY='your-key-here'    # Windows: set GROQ_API_KEY=your-key-here
```

The two LLMs used in this project (`gpt_oss_20b` and `gpt_oss_120b`, see `src/llm_client.py`)
are both served through Groq's free tier, so no local model downloads or GPU are required.

## Workflow (run in this order)

1. **Fill in `data/questions.csv`** — this is manual work: write/collect your hearing-health Q&A pairs with ground truth (see schema in the file header).
2. **Drop source documents into `data/corpus/`** — download hearing health PDFs/pages from WHO, CDC, NIH, ASHA.
3. Run `python src/01_build_corpus.py` — chunks documents at 200/500/1000 tokens, saves to `data/chunks/`.
4. Run `python src/02_embed_and_index.py` — builds a Chroma vector store for each (embedding model × chunk size) combination.
5. Run `python src/03_run_experiment.py --grid pilot` (or `--grid full`) — loops through every configuration in `config.py`, queries each question, logs results to `results/run_log.csv`.
6. Run `python src/04_score_answers.py --mode validate` first to spot-check the judge against human labels, then `python src/04_score_answers.py --mode score` to score the full run.
7. Open `notebooks/eda_starter.py` — explore your question set and corpus before diving into stats.
8. Run `python src/05_analyze_results.py` — runs the statistical tests and saves figures to `figures/`.

## Notes
- Start small: test the full pipeline end-to-end with a handful of questions and the `pilot` grid (4 configs) before scaling to the full factorial grid (`full`, 24 configs). This catches bugs cheaply.
- Both LLMs run through Groq's free tier — budget time for rate limits/retries on larger runs (see `llm_client.py` for retry/backoff handling).
- `data/corpus/`, `data/chunks/`, `data/vector_stores/`, and files in `results/`/`figures/` are generated by the pipeline and are not tracked in version control — see `.gitignore`. Re-run the steps above to regenerate them locally.
